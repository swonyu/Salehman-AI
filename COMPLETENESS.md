### ⬜ #1 — No loss-limit / tilt circuit breaker that HALTS trading. Verified: grep for lossLimit/dailyLoss/halt/circuit across StockSage returns zero hits; StockSageTradeGate.evaluate (StockSageTradeGate.swift:40) is per-trade only and never sees recent results. The journal already records realizedProfit/realizedR/closedAt on every TradeRecord (StockSageJournal.swift:59-60) and the consecutive-loss streak (streakSummary, Journal.swift:565) — the data to slam the brakes exists, but nothing acts on it. This fuses the daily/weekly realized-loss limit AND the loss-streak cooldown into ONE behavioral guard (they are the same circuit breaker measured two ways: cumulative R and consecutive count).
**signature:** struct LossLimitPolicy: Sendable, Equatable { var maxDailyLoss: Double?; var maxWeeklyLoss: Double?; var maxDailyLossR: Double?; var maxWeeklyLossR: Double?; var standDownLossRun: Int = 3; var warnFraction: Double = 0.70 }
struct LossLimitState: Sendable, Equatable { enum Status: String, Sendable { case ok, warn, halted }; let status: Status; let dailyRealized: Double; let weeklyRealized: Double; let dailyRealizedR: Double; let lossRun: Int; let haltReason: String? }
enum StockSageLossLimit { /// Pure: aggregate today's/this-week's realized P&L+R AND consecutive-loss run from closed TradeRecords vs the policy. Calendar+now injected for determinism. nonisolated static func evaluate(closedTrades: [TradeRecord], streak: JournalStreak?, policy: LossLimitPolicy, now: Date, calendar: Calendar = .current) -> LossLimitState }

**whyItCosts:** This is the single mechanism that actually blows up retail accounts: revenge/over-sizing after a red run. The app meticulously sizes every trade to 1% but has zero floor on how many losers in a row the owner can keep clicking. A -2R morning becomes a blown week with full app blessing because 'this trade' always passes the per-trade math.

**test:** @Test dailyDollarLossHalts() — two losers today past maxDailyLoss => .halted, haltReason mentions 'daily'. @Test threeLossStreakStandsDown() => .halted. @Test warnBandAtSeventyPercent(). @Test yesterdaysLossExcludedFromToday() (closedAt = now-26h). @Test openTradesIgnored() (closedAt==nil contributes 0). @Test breakevenScratchDoesNotCountAsLoss() (R==0 breaks the run, matching StockSageJournal.streak). @Test profitableDayIsOk().

**caveat:** Behavioral brake, NOT a probability edge — a losing streak does not make the next trade likelier to lose; markets have no memory. It only sees trades the owner LOGS (trades outside the journal are invisible) and only CLOSED trades count toward today's R — an open loser isn't counted until closed. Period reset is calendar-based in local time, not a timer.

### ⬜ #2 — No account-level roll-up that the pre-trade gate consults. StockSageTradeGate.evaluate (StockSageTradeGate.swift:40) checks only this trade's stop/risk%/R:R/one pairwise correlation/earnings — it has no parameter for portfolio heat, exposure-cap breach, the loss-limit halt, or the market regime. StockSageRegime.assess already classifies .crisis/.trendingBear/.ranging (Regime.swift:18-22) and drives SIZING bias but never produces a GO/NO-GO at the gate. Extend evaluate with defaulted-nil params so every existing caller and test is byte-unchanged.

**whyItCosts:** The owner's mental model is 'the app told me this trade was clear.' This is the seam that lets rank 1, 6, and 7 actually reach the go/no-go moment.

**caveat:** Pure aggregation — no new computation, it consumes the structs ranks 1/6/7 produce. Regime is a LAGGING, market-WIDE gauge (S&P vs 200DMA + VIX + breadth) that knows nothing about the specific symbol, and a crisis is sometimes exactly when the best R:R appears — so .crisis is a default-cautious posture (downgradable to a warn), never an auto-block, and it must never fabricate a regime when the keyless Yahoo feed is off (degrades to 'no vote').
