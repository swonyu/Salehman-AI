### ✅ DONE #1 [high] — "inf"/"infinity" in Acct $ or Risk % field crashes the app via Int(.infinity) in the position sizer (persists across relaunch)
**fix:** journalField (MarketsView.swift:1442-1448) is a plain unfiltered TextField bound directly to @AppStorage String "marketsSizerAccount"/"marketsSizerRiskPct" (1431-ish; bindings at 2830/2832, declared 68-69). The detail-sheet sizer at 2834 does `if let acct = Double(sizerAccount), let rp = Double(sizerRiskPct)` with NO `acct > 0` or `.isFinite` filter and feeds StockSagePositionSizer.size. size() guards only `account > 0, riskFraction > 0, entry > 0, stop > 0` (StockSagePositionSizer.swift:24) — never .isFinite. Double("inf")/Double("infinity") parse to +Infinity, which is non-nil and passes `> 0`, so riskBudget=∞ reaches `Int((riskBudget/riskPerShare).rounded(.down))` (line 28) → Int(.infinity) is a Swift runtime trap → hard crash. Same unguarded raw-Double→size path at the copy-plan button 3250-3251 and the what-if site 3156-3157. The corrupt value persists in UserDefaults, so the app re-crashes on every relaunch the moment an idea-detail with a stopPrice is shown (the sizer block is unconditional once `idea.advice.stopPrice != nil`). Repro: type `inf` into the "Acct $" field on any buy-rated idea detail that has a stop; or set UserDefaults marketsSizerAccount="inf" and reopen any idea with a stop. Fix: replace the raw `Double(sizerAccount)`/`Double(sizerRiskPct)` parses at 2834, 2391-2392, 2516, 2629, 3156, 3250 with the existing validators `StockSageInput.positiveAmount(sizerAccount)` / `StockSageInput.percent(sizerRiskPct)` (StockSageInput.swift:17-26, which enforce .isFinite/>0/cap). Belt-and-suspenders: add `guard account.isFinite, riskFraction.isFinite, entry.isFinite, stop.isFinite else { return nil }` to StockSagePositionSizer.size after line 24, and guard the Int conversion: `let raw=(riskBudget/riskPerShare).rounded(.down); guard raw.isFinite, raw>=0, raw<=Double(Int.max) else { return nil }`.

### ⬜ #4 [low] — Ollama server URL accepts a schemeless host and silently goes mute (no scheme prepend, no validation)
**fix:** ollamaBaseURLCurrent (AppSettings.swift:256-261) only trims whitespace and strips trailing slashes; it never ensures an http:// scheme. Typing the Tailscale address as `100.111.178.2:11434` (natural host:port form) produces `URL(string: "100.111.178.2:11434/api/version")` with host=nil — a URL scheme must begin with an ASCII letter, so "100.111.178.2" can't be a scheme and Foundation parses it as a degenerate path-only relative reference. isUp() (OllamaClient.swift:60) and hasModel() (75) then build requests that never reach the server; the brain silently reports unreachable with no diagnostic, even though the host:port was correct. NOT a crash (all sites guard `URL(string:)`), NOT a secret leak, NOT an auto-spend — Ollama paths are keyless (only Content-Type, no Authorization). Usability robustness gap. Repro: Settings → Ollama server URL = "100.111.178.2:11434" (no http://); brain reports unreachable. Fix: in ollamaBaseURLCurrent, after trimming, prepend a scheme when absent: `if !s.isEmpty, !s.lowercased().hasPrefix("http://"), !s.lowercased().hasPrefix("https://") { s = "http://" + s }`. Optionally surface a red hint when the normalized URL has a nil host.

### ⬜ #5 [low] — Reachability cache (30s TTL) not invalidated on Ollama URL change — availability gate lags the new server up to 30s
**fix:** OllamaClient.Reachability (OllamaClient.swift:50-92) caches isUp() (upUntil/upValue) and the model list (modelsUntil/modelNames) for 30s and stores NO URL — it's only invalidated inside warmUncensoredIfSelected (the invalidate() at line 89-91 is called from there). AppSettings.ollamaServerURL's didSet (AppSettings.swift:78-79) persists the new value but does NOT invalidate the cache. So after the owner switches the server URL (localhost ↔ 100.111.178.2 over Tailscale), isUp()/hasModel() — and the brain-availability gate that reads them — can return the PREVIOUS server's up/down + model list for up to 30s, wrongly blocking or admitting a request. Self-heals within 30s; `base` is a live computed property (line 45) so generate()/chatStream() hit the new server immediately once admitted — only the reachability/model gating lags. Not a sacred-rule violation (keyless/local, no spend, no leak). Repro: send a message (warms the cache vs localhost) → change Settings Ollama URL → send again within 30s; the gate still reflects localhost. Fix: in ollamaServerURL didSet, add `Task { await OllamaClient.invalidateReachability() }` exposing a static wrapper around Reachability.shared.invalidate() (mirroring the existing warmUncensoredIfSelected call), or store the base inside Reachability and treat a changed base as an automatic cache miss.
