# Newest-engine bughunt (wrnwv4smr, 2026-06-22)

10 findings. #1/#3/#4/#5 FIXED. #2 (wire GapRisk+LossLimit into MarketsView UI) DEFERRED — real, UI scope.

### ✅ DONE #1 [high] GapRisk `leverage` is validated and stored but never applied to the loss math — leveraged scenarios understate the loss and the "can exceed your account" warning fails to fire
**fix:** The whole point of this engine is to warn that 'with leverage the loss can be MORE than your entire account.' Yet `leverage` is guarded (line 43) and copied into GapRiskScenario (line 55) while NO formula references it: dollarsLost = shares*lossPerShare (line 49), accountLossPct = dollarsLost/accountEquity (line 56), beyondPlanDollars, rMultiple, exceedsAccount are all leverage-independent. Pick ONE: (a) APPLY it — `let dollarsLost = shares * lossPerShare * leverage`, and let accountLossPct/exceedsAccount/beyondPlanDollars follow; OR (b) if `shares` is intended to already be the gross/levered share count, REMOVE the `leverage` parameter, its guard, the struct field, and the worstCase/fromPosition plumbing so the API stops implying an amplification it doesn't perform. Do not ship a named, validated knob that silently no-ops while the caveat string (line 34) and verdict (line 27) promise it drives the account-blowup case.

### ⬜ UI-wiring #2 [high] GapRisk and LossLimit are never surfaced anywhere — no UI, no tool, no store wiring — so their mandatory caveats reach no user
**fix:** A repo-wide search (excluding the two engine files) returns ZERO references to StockSageGapRisk, GapRiskScenario, exceedsAccount, blowsThroughStop, StockSageLossLimit, LossLimitState, LossLimitPolicy, or standDown — confirmed empty. MarketsView.swift renders many `.caveat` surfaces but none for gap risk or the loss-limit breaker; StockSageStore.swift publishes neither. Wire both into the Markets UI behind their existing caveat strings: (1) a GapRisk 'a stop is not a fill' table on the position-sizer/trade-plan surface rendering each worstCase(...) scenario's `verdict` plus StockSageGapRisk.caveat; (2) surface LossLimitState (status + haltReason + caveat) on the journal/money-velocity summary so a halted run actually shows. Add a UI/integration assertion (mirroring the existing caveat-presence sweep) that both caveat strings are reachable from a rendered view so this can't silently regress.

### ✅ DONE #3 [medium] LossLimit loss-run silently skips closed losers whose stop==entry (realizedR is nil), so a real money loss neither counts toward nor breaks the streak
**fix:** The streak is built from `closed.compactMap { guard let c = t.closedAt, let rr = t.realizedR else { return nil }; ... }` (line 64-65). realizedR (journal line 60 -> rMultiple line 52-56) is nil whenever riskPerShare==0 (stop==entry). Such a trade can still be a real cash loss (realizedProfit non-nil/negative — profit(at:) needs no stop). compactMapping it out makes it invisible: it neither increments lossRun nor breaks it, so an older win can wrongly appear 'most recent' (resetting the streak) or two losses straddling it get treated as consecutive. This is inconsistent with the dollar gate, where the same trade DOES contribute its negative dollars (line 56). Decide the streak on realized P&L (or whichever of R/P&L is available): build `recent` from any closed trade with a non-nil realizedProfit, treat profit<0 as a loss and profit>=0 (incl. exactly 0) as a break, falling back to R only when profit is unavailable.

### ✅ DONE #4 [medium] LossLimit calendar defaults to .current — daily/weekly halt windows drift by timezone, locale firstWeekday, and DST, disagreeing with the journal's own UTC-pinned aggregation
**fix:** `evaluate(... calendar: Calendar = .current)` derives dayStart = startOfDay(now) (line 49) and weekStart from dateInterval(of:.weekOfYear) (line 50). With .current the day boundary shifts with device TZ/DST and the week boundary shifts with locale firstWeekday (Sun in US vs Mon in ISO/EU) — so the SAME closed trades produce different halt verdicts on two machines or after travel, and the journal's OWN reporting deliberately pins a fixed UTC Gregorian calendar for exactly this reason (StockSageJournal.swift ~line 507 and ~511-512: `cal.timeZone = TimeZone(identifier: "UTC")!`). The safety breaker is the one place that does NOT pin it. Pin the boundary calendar the same way — default to a fixed Gregorian/UTC calendar (or require callers pass the journal's calendar so all aggregations agree). If local-time resets are genuinely wanted (the line 42 caveat says 'local time'), at minimum set an explicit firstWeekday so the weekly window can't change with locale, and document the 23/25h DST days.

### ✅ DONE #5 [medium] CapitalAllocator bypasses the 20% per-position Kelly cap — a single idea can carry up to 50% account risk when requestedHeat <= maxHeat
**fix:** Line 60-62 build the weight from k.halfKelly (the RAW fraction fStar/2 in [0,0.5]) instead of k.suggestedFraction, which StockSageKelly hard-caps at maxFraction=0.20. The heat invariant still holds (the uniform scale pins Sum <= cap), so this is NOT a cap breach — but when requestedHeat <= cap (one strong idea + a generous maxHeat, e.g. the test's 0.50) no scaling occurs and a lone position can sit at up to 50% of the account at its stop, the exact over-betting the standalone Kelly sizer's 20% cap exists to prevent. Either clamp each raw weight before summing — `let raw = Swift.min(StockSageKelly.maxFraction, k.halfKelly)` — so the allocator inherits the same per-idea discipline as everywhere else Kelly is used; OR, if the unclamped edge-proportional weighting is deliberate, add a one-line comment at line 60 stating the allocator intentionally drops the per-idea 20% cap and relies solely on the heat cap, so a future reader doesn't 'fix' it.

### ⬜ #6 [low] GapRisk side-correctness is assumed, not guarded — a mislabeled long (stop>entry) yields rMultiple=-1 and a phantom 'gain' at gapPct==0, breaking the rMultiple>1 invariant
**fix:** riskPerShare uses abs(entry-stop) (line 41), so the guard passes for a 'long' with stop ABOVE entry (or a 'short' with stop below). The sign math then breaks: long entry=100/stop=105/gap=0 gives lossPerShare=entry-gapFill=100-105=-5, rMultiple=-1.0, dollarsLost negative (a phantom gain), and the documented 'rMultiple>1 exactly when gapPct>0' invariant fails. Add a side-consistency guard so malformed inputs return nil: in the switch (or before it) require `case .long: guard stop < entry else { return nil }` and `case .short: guard stop > entry else { return nil }`. This makes the rMultiple invariant unconditional and prevents negative-loss scenarios.

### ⬜ #7 [low] GapRisk worstCase() does not sort `gaps`, so the documented 'ascending in loss' ladder only holds for already-sorted input
**fix:** The doc comment and test assert the ladder is 'ascending in loss' and dollarsLost is strictly monotonic in gapPct (lossPerShare = riskPerShare + stop*gapPct). But worstCase maps over `gaps` in caller order without sorting (line 65), so a custom unsorted array (e.g. [0.20,0.05,0.35]) produces a non-ascending result, silently breaking the contract; the test only exercises the default [0.05,0.08,0.20,0.35]. Sort by magnitude before mapping: `gaps.sorted().compactMap { scenario(... gapPct: $0 ...) }`. Since loss is monotonic in gapPct, sorting guarantees ascending dollarsLost for any caller ladder.

### ⬜ #8 [low] CapitalAllocator sort is non-deterministic for duplicate/mixed-case symbols at equal riskFraction — defeats the file's 'Pure + deterministic' contract
**fix:** Line 85 sorts by riskFraction desc with symbol asc as the ONLY tie-break. Swift's Array.sorted(by:) is not guaranteed stable, so two AllocatedPositions with equal riskFraction AND equal symbol (or symbols differing only in case — '<' compares raw strings, so 'btc' vs 'BTC' don't tie but genuine duplicates do) get unspecified relative order, breaking the 'Pure + deterministic' contract (line 10) and the struct doc (line 25). riskFraction collides whenever two ideas share raw half-Kelly (same conviction + R:R — common). The allocator does no symbol dedup. Add a total-order final tie-break so the comparator is a strict weak ordering with no ties: e.g. `(riskFraction desc, symbol asc, dollarsAtRisk desc, notional desc)`, or normalize/merge duplicate symbols on entry.

### ⬜ #9 [low] LossLimit weekly window silently collapses to the single current day when dateInterval returns nil — the weekly gate would fail open
**fix:** `let weekStart = calendar.dateInterval(of: .weekOfYear, for: now)?.start ?? dayStart` (line 50). If dateInterval ever returns nil, the weekly aggregation window collapses to [startOfToday, now], so maxWeeklyLoss/maxWeeklyLossR quietly under-report and fail to halt on a genuinely bad week — a guardrail failing open with no signal and the wrong scope. For valid Gregorian dates this effectively never returns nil, so the likelihood is low; the concern is the silent wrong-scope fallback. Fall back to a true 7-day window instead — `calendar.date(byAdding: .day, value: -6, to: dayStart)` or `now.addingTimeInterval(-7*86400)` — so the weekly gate never silently shrinks to one day. Alternatively assert/log if it is ever nil rather than degrade silently.

### ⬜ #10 [low] GapRisk has no upper bound on gapPct — a long with gapPct>1 produces a negative gapFillPrice (a physically impossible sub-zero price)
**fix:** Guards reject gapPct<0 but allow arbitrarily large gapPct (line 42). For a long, gapFill = stop*(1-gapPct) goes negative once gapPct>1 (gapPct=1.5, stop=95 -> gapFill=-47.5), implying the stock traded below $0. dollarsLost/rMultiple stay finite and monotonic (no divide-by-zero), but a negative gapFillPrice is a degenerate scenario the 'rejects degenerate inputs' contract arguably should catch; shorts have no analogous ceiling so the asymmetry only affects the long branch. Optional hardening: for the long branch clamp gapFill at 0 — `gapFill = max(0, stop*(1-gapPct))` (a 100%+ gap down is a total wipeout, not a negative price) — or add a guard rejecting gapPct>1 for longs.
