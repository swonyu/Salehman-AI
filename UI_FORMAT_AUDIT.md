### ✅ DONE #1 [medium] — Per-share cost basis renders sub-penny holdings as "@ 0.00" — fabricates a zero-cost (free) position
**file:** /Users/saleh/Desktop/Salehman AI/Salehman AI/Views/MarketsView.swift
**fix:** Confirmed at MarketsView.swift:767 — `Text("\(numString(p.shares)) sh @ \(String(format: "%.2f", p.costBasis))")`. costBasis is validated only as `costBasis >= 0` (StockSagePortfolio.swift:42) and the add form is free-form; the universe ships sub-$1 names (DOGE-USD, XRP-USD, ADA-USD at QuoteService.swift:290) plus user-enterable micro-caps, so a $0.0023 basis is reachable and rounds to "0.00", reading as a free position (infinite return over $0). This is the sharpest single misleading number because it directly corrupts the P&L model. Fix: replace the fixed %.2f with adaptive precision for sub-dollar magnitudes — when abs(price) < ~1, widen to enough significant decimals (e.g. `.formatted(.number.precision(.significantDigits(2...6)))` or a magnitude-keyed decimal count) so $0.0023 shows "@ 0.0023". Apply the identical adaptive helper to the per-share live value at line 772 (`String(format: "%.2f", value)`), which has the same flaw. Use one shared formatter so both rows stay consistent.

### ⬜ #2 [medium] — Rebalance panel sums mixed-currency holding values un-converted, then labels the trade sizes "$"
**file:** /Users/saleh/Desktop/Salehman AI/Salehman AI/Views/MarketsView.swift
**fix:** Confirmed: `rebalHoldings` (MarketsView.swift:846-847) is built from the LOCAL-currency `holdingValue(...)` with NO `* rate`, unlike `portfolioTotals` (609-610) which multiplies by `fxRatesToUSD`. `StockSageRebalance.plan` then does `total = holdings.reduce(...)` (StockSageRebalance.swift:33) summing GBP+USD+SAR+JPY at 1:1, and `deltaValue = drift * total` (line 50) — whose doc field asserts "account currency" (line 16) — inherits the cross-currency total. The UI stamps it as dollars: header line 857 and `String(format: "$%.0f", abs(t.deltaValue))` line 860. Net effect: a fake dollar trade size, and skewed 1:1 weights can suppress a genuinely-needed rebalance. Fix: convert each holding to USD before feeding the plan, mirroring portfolioTotals — build rebalHoldings as `holdingValue(...) * (fxRatesToUSD[StockSageCurrency.currencyForSymbol($0.symbol)] ?? skip)`, dropping untracked-FX holdings (as line 607 does). Then the "$" labels at 857/860 are truthful. Route the allocationPanel holdings (line 906) and the what-if/sizing path (line 3273) through the same per-holding `* rate` so their slice percentages aren't distorted by summing currencies 1:1.

### ⬜ #3 [medium] — StockSageLiquidity.humanDollars: K and M tiers render band tops as "$1000K"/"$1000M" instead of "$1.0M"/"$1.0B"
**file:** /Users/saleh/Desktop/Salehman AI/Salehman AI/StockSage/StockSageLiquidity.swift
**fix:** Confirmed at StockSageLiquidity.swift:73-80. The K branch (line 77) `String(format: "$%.0fK", v/1_000)` and the M branch (line 76) `String(format: "$%.0fM", v/1_000_000)` pick the tier on raw v but format with %.0f, whose half-up rounding pushes the band top over the next threshold AFTER the tier was chosen: v in [999_500, 999_999] prints "$1000K" (should be "$1.0M"); v in [999_500_000, 999_999_999] prints "$1000M" (should be "$1.0B"). This is real avg-dollar-volume money labeling. Fix: promote on the rounded magnitude — after computing the divided value at the tier's precision, if it rounds to >= 1000 step up to the next unit. Cleanest is a single round-then-pick-unit helper covering all tiers; ranked above RSFormat.gp because this labels real dollars.

### ⬜ #4 [medium] — RSFormat.gp: K and M tiers render band tops as "1000.0K"/"1000.00M" instead of "1.0M"/"1.00B"
**file:** /Users/saleh/Desktop/Salehman AI/Salehman AI/Views/RuneScapeMarketView.swift

Ideally share the same round-then-pick-unit helper as humanDollars.
