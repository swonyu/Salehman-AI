### ❌ REJECTED #5 [medium] Backtester charges round-trip cost on one side only — risk denominator (entry−stop) is never widened, unlike NetEdge
**fix:** The doc says costs charge 'round-trip ... against EVERY trade's R', but `r = (exitPrice - entry - costPerShare) / risk` only shrinks the numerator while `risk = entry - stop` stays gross — inconsistent with NetEdge (StockSageNetEdge.swift:65-66) which does `netReward = grossReward - cost; netRisk = grossRisk + cost`. Preferred fix (a): widen the denominator to match NetEdge — `let r = (exitPrice - entry - costPerShare) / (risk + costPerShare)`. This makes per-trade R cost-consistent across the app and makes stop-outs slightly worse than −1R (cost-honest). Alternative (b) if single-sided is intentional: correct the doc at :81-85 and the header to say 'costs charged against realized reward only, not round-trip against R'. Recommend (a).
**test:** XCTest: a stopped-out trade with a positive cost must net strictly worse than −1.0R. Python-verifiable for fix (a): `def r_costed(exit, entry, stop, cost): return (exit-entry-cost)/((entry-stop)+cost)`; a stop fill (exit==stop) with entry=100, stop=98, cost=0.1 -> `r_costed(98,100,98,0.1)` ≈ -2.1/2.1 = -1.0476 < -1.0 (assert < -1.0); and a 2R winner (exit=104) -> `r_costed(104,100,98,0.1)` ≈ 3.9/2.1 = 1.857 < 2.0 (assert winner shrinks vs gross 2.0). Compare to current single-sided which gives the stop at exactly -1.05 over gross risk 2.0 = -1.05 — assert the new value differs and is the more conservative magnitude.

### ⬜ #8 [low] 95% VaR uses nearest-rank percentile, systematically biased toward the tail on the small samples this runs on
**fix:** percentile() uses nearest-rank `idx = Int((N−1)*0.05).rounded()`, which for typical overlapping-history sizes collapses toward the single worst day (N=8→idx0 ≈12.5% tail; N=50→idx2 ≈6%) and overstates the labeled '95% VaR'. Switch to linear-interpolation percentile: `let pos = Double(sorted.count - 1) * p; let lo = Int(pos.rounded(.down)); let hi = Int(pos.rounded(.up)); let frac = pos - Double(lo); return sorted[lo] + frac * (sorted[hi] - sorted[lo])`. This keeps endpoints intact (p=0→min, p=1→max) and removes most small-N bias. Optionally also gate VaR display behind observations >= 30, else show n/a.
**test:** XCTest endpoints unchanged (`percentile(v,0)==min`, `percentile(v,1)==max`) plus a mid-case. Python-verifiable against numpy's linear interpolation: `import numpy as np; v=[-0.05,-0.03,-0.02,0,0.01,0.02,0.03,0.04]; assert abs(interp_percentile(v,0.05)-np.percentile(v,5))<1e-9` where interp_percentile is the formula above; assert it differs from the nearest-rank result `sorted(v)[round((len(v)-1)*0.05)]` (which returns the single worst, -0.05).
