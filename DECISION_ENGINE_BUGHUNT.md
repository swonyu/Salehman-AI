# Decision-engine bughunt (wptj7zxb1, 2026-06-22)

3 CONFIRMED bugs (survived 2-skeptic refute). #1 (unbounded rewardR overruns regime ban) DONE. #2/#3 are real advisor-logic gaps — RE-VERIFY vs source.

### ✅ DONE #1 [high] ev-ranking — Unbounded rewardR lets a regime-banned side defeat the −1,000,000 gate and rank #1
**problem:** ev() computes rewardR = reward/risk with risk = abs(entry-stop) and guards only `risk > 0` (line 71); there is no floor on stop distance, so rewardR, evR and the base ranking key qualityAdjustedEVR are UNBOUNDED. The regime gate at line 152 demotes a banned side by a FIXED -1_000_000 and the comment at 130-131 asserts this 'always ranks below every non-banned idea' — an invariant that only holds while base stays small. A hair-thin stop makes base >> 1,000,000, so a regime-banned idea clears every penalty (regime -1,000,000, cost -500,000, conviction -1000, velocity -1000) and ranks above legitimate ideas — exactly what bannedFromTopRank exists to prevent. Reproduces through the public rankByEV(_:regime:) API with hand-built StockSageIdea values.

**failingInput:** EV.rankByEV([cleanBuy, bannedSell], regime: MarketRegime(state: .trendingBull, riskScore: 0.6, signals: [], sizingBias: 1.1, caveat: "x")). cleanBuy = idea price 100, action .buy, conviction 1.0, stop 90, target 120 (rewardR 2, base key 0.74). bannedSell = idea price 100, action .sell, conviction 1.0, stop 100.00001 (risk 1e-5), target 80 (reward 20 -> rewardR 1,999,999.99). evR = p*rewardR-(1-p) with p=0.58 -> 1,159,999.58; qualityWeight=1.0 -> base 1,159,999.58. regimeAdjustedEVRankKey demotes the banned SELL by only -1,000,000 -> +159,999.58, which exceeds cleanBuy's 0.74. Result: .first?.symbol is the banned SELL in a trendingBull tape (a short crowned #1 in a bull). clearsCostAfterFrictions does NOT save it: with risk~1e-5 the net break-even win rate is ~0, so clearsCost returns true and the -500,000 cost penalty never fires. Same unbounded base also overruns the -1000 low-conviction band (line 124) and the -1000 velocity band (line 158).

**fix:** Bound rewardR before it enters the ranking math so a fixed demotion constant can always dominate it. Inside ev() (StockSageExpectedValue.swift:72) cap the ratio: `let rewardR = Swift.min(reward / risk, 50)` (50:1 is already absurd for display, and keeps base well under the 1,000,000 / 500,000 / 1000 penalties). Optionally also floor the stop distance (return nil when risk < entry * minRiskFraction, mirroring the entry==stop nil guard) so degenerate setups are dropped, not merely capped. A robust alternative is to make the regime/cost/conviction gates RELATIVE — partition ideas into banned/non-banned tiers first and sort by base within each tier — instead of relying on fixed magnitudes against an unbounded key.

### ⬜ #2 [medium] advisor-score — Range-bound regime can emit Strong Buy / Buy with a full trade plan, contradicting the engine's own 'avoid chop' design
**problem:** regime is forced to .range whenever er < 0.30 (trending=false, line 181), but `action` is derived purely from the raw `score` switch (lines 186-192) with no regime gate. The +0.40 trend term, +0.15 momentum and +0.10 MACD are all added before the regime is considered, so a noisy-but-rising name can reach score >= 0.5 while er < 0.30. Result: a Strong Buy (or Buy) card tagged 'Range-bound' with a stop, target and position size attached — directly contradicting the documented design at lines 183-184 ('In a choppy regime with no edge, prefer Avoid ... forcing trades in chop loses') and the Action.avoid doc at line 15 ('choppy / no edge — stand aside').

**failingInput:** A ~210-bar series: a smooth uptrend for the first ~190 bars (e.g. 50 + i*0.5) then ~20 bars of large zig-zag (deltas like +8,-7,+8,-7,... netting slightly up). Verified numerically: price=162.5 > sma50=143.57 > sma200=105.08 (+0.40), 126-bar mom=+77.6% (+0.15), MACD histogram>0 (+0.10) -> score≈0.65. But efficiencyRatio over the LAST 20 bars = 0.125 < 0.30 -> trending=false -> regime=.range. trendOK is true (bullish name) so the 12-1 veto does not fire. advise() returns action=.strongBuy with regime=.range, plus a stop/target/suggestedWeight (isBuy path, lines 197/212).

**fix:** Gate the buy family on regime after computing `action` (around StockSageAdvisor.swift:192): `if !trending && (action == .strongBuy || action == .buy) { action = .hold }` — never issue a trend-following long in a regime the engine itself labels no-edge chop. Stronger version: in a non-trending regime, only allow a buy when there is a mean-reversion trigger (RSI < 30 oversold bounce already scored at line 130) and never allow .strongBuy. The regime should constrain the action, not just label the card.

### ⬜ #3 [low] advisor-score — The 126-bar momentum signal is triple-counted (raw momentum + relative strength + vol-adjusted momentum), inflating conviction from one move
**problem:** Three additive score terms are all driven by the SAME 126-bar return: mom (±0.15), relativeStrength = symRet-benchRet (±0.08), and volAdjustedMomentum = mom/atrPct (±0.05, same sign as mom). When the benchmark is roughly flat, rs ≈ symRet ≈ mom, so rs is not independent — it re-scores the drift mom already scored; vam is definitionally the same sign and source as mom. The comments call each term independently 'additive', but they are collinear, so a single 126-bar move can stack +0.28 and systematically over-convert marginal momentum into Buy/StrongBuy while overstating conviction. (Lower money impact than #1/#2: it requires the optional benchmark + highs/lows inputs, and rs/vam do carry marginal information, but the additive treatment double/triple-counts one fact.)

**failingInput:** advise(closes:, highs:, lows:, benchmarkCloses:) for a name +20% over the last 126 bars in a flat market (benchmark ~0% over the same 126 bars): mom>0 -> +0.15 (line 116); relativeStrength = symRet - benchRet ≈ +20 (>0) -> +0.08 (line 169); |mom/atrPct| >= 5 with highs/lows present -> +0.05 (line 158). One 6-month up-move contributes +0.28 to score (over half the 0.5 StrongBuy threshold) and raises conviction (min(|score|,1)) correspondingly, before any trend/MACD term. Confirmed all three use the same 126-bar window: mom = returnOverPeriod period min(count-1,126); relativeStrength default period 126; volAdjustedMomentum default period 126 (StockSageIndicators.swift:163,174).

**fix:** Make the momentum family share a budget instead of summing independently. Compute one momentum sub-score from mom, then let relativeStrength only ADJUST it (small bonus/penalty for out/under-performing the benchmark) and treat volAdjustedMomentum as a QUALITY scaler on the mom term rather than an additive term. Minimum viable fix: clamp the combined mom+rs+vam contribution, e.g. accumulate them into a local `momentumContribution` and add `Swift.max(-0.20, Swift.min(0.20, momentumContribution))` to score.
